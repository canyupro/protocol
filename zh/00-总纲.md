## 一、本规程是什么

并行开发规程是一套**对话驱动的多 AI 协作开发方法**。它规定了从一个空白目录到项目发版的完整过程，覆盖三个规程域：

- **完整开发流程**：Round 0-8 串行→并行→收尾，含前端和 AI Agent 工程节点
- **全业务开发范式**：CRUD + 前端（完成定义/视角收束/单源View）+ AI Agent（RAG/记忆/SSE/降级/Prompt 版本化/危机短路）+ 异步分工（调度方/执行方/执行方先行）
- **特殊机制补充**：约束介质分层（L0/L1/L2）+ 继承介质（分叉/子智能体/快照）+ 介入机制（升级四档）+ MCP tool + 声明卡 + 验收锁 + 环境 Runbook

三个核心要素：

- **方法论层**：三段式流程、渐进冻结、三轨分级
- **对话协议层**：议题清单、决策记录、逐轮确认、模式声明卡、交付一致性验收锁
- **操作规范层**：冻结判定、触发器、模板、检查清单、环境 Runbook、前端完成定义、MCP tool 接口、继承快照

## 二、核心哲学

### 2.1 三条基础哲学

| 原理 | 表述 | 表现 |
|------|------|------|
| Token 换时间 | 1 个协调方主导思路 + N 个 AI 并行执行 | Round 6 链路多 Agent 同时推进 |
| Day 0 vs Day 10 4 倍返工 | 框架必须完全冻结才能并行 | Round 0-5 冻结审计 + 启动门硬门槛 |
| 细节可委托，验证不可委托 | 协调方必须亲自跑动态测试 | Round 6-8 强制 pytest 真跑 + 5 维度审计维度 2 |

### 2.2 三条范式转换

#### 范式一：约束介质从文本层迁到协议/工具层

文本是软约束，模型可软违反；tool call 是硬约束，模型再弱也得调用。

| 层 | 介质 | 强约束度 |
|---|---|---|
| L0 文本 | 规则文档 / 触发器 / 模板 | 低（软约束） |
| L1 协议 | 模式声明卡 / 产出契约 / 验收 checklist | 中 |
| L2 工具 | MCP tool（不调用不能产出） | 极高（硬约束） |

迁移判据：违反后果严重 + 文本层反复失效 → 迁往 L2。违反后果中等 + 需跨轮次约束 → 迁往 L1。其余留 L0。详见 06 号文件。

#### 范式二：度量从规程动作计数迁到价值/工时比

代码 LOC / 决策计数 / 候选条款数 / commit 数全是噪声。**价值/工时比（用户可见实质能力增量）是唯一真指标。**

判据：轮次成立的判据是「用户可见的实质能力增量」，不是「走了规程规定的步骤」。

#### 范式三：制衡从自己触发迁到角色分离

自己解释、自己执行的规则等于没有。执行方不写规则，写规则方不自己触发。制衡条款的触发权必须交给非执行方。详见 06 号文件。

## 四、三条轨道

### Demo Mode（极速演示模式）

准入条件（四条硬门槛，缺一不可）：
- 目的声明：做完后不会上线运营，仅用于演示/验证概念
- 无用户数据沉淀：SQLite 或内存即可
- 不需要多 Agent 并行：你一人，或最多 1 个编码 Agent
- 生命周期不超过 2 周

压缩为 4 轮对话，跳过启动门、跳过 5 维度审计、跳过 E2E 测试。

### Standard Track（默认轨道）

所有项目默认走此轨道。串行对话 9 轮 + 并行 1 轮 + 收尾 2 轮 = 共 11 轮。适用于 2-4 条链路、中低耦合、不超过 2 人的团队。

### Premium Track（完整轨道）

在 Round 2b 结束时由协调方主动提问，根据以下 6 项指标判断是否升级：

| 判定项 | Standard | Premium |
|--------|----------|---------|
| 链路数 | 2-4 条 | 5 条及以上 |
| 链路间耦合度 | 低耦合 | 高耦合 |
| 技术栈复杂度 | 单后端 + 单一数据库 | 微服务 / 多数据库 / AI 模型 |
| 团队规模 | 不超过 2 人 | 3 人及以上 |
| 需求稳定性 | 稳定 | 不断变化 |
| 合规/安全要求 | 无特殊 | 有合规审计需求 |

3 项及以上滑向 Premium 时可以决定升级。

## 七、三档冻结判定规则

判断一项决策是硬冻结、软冻结还是草稿，依次回答三个问题：

**Q1**：如果这项决策被推翻，是否需要修改 2 个及以上已冻结的产出物？
是 → 硬冻结。否 → 继续 Q2。

**Q2**：如果这项决策被推翻，是否会改变 2 个及以上 Agent 的白名单边界？
是 → 硬冻结。否 → 继续 Q3。

**Q3**：这项决策是否属于架构级（目录结构/跨模块协议/语言和框架）？
是 → 硬冻结。否 → 软冻结。

草稿：本轮新产出、尚未经过确认的内容。

**默认策略**：模糊时一律标软冻结，冻结审计轮再决定是否升级。软冻可升硬（审计轮加固），硬冻降软需要走回滚协议（代价大）。

**每个硬冻结决策必须附带**："可被推翻的条件"字段。

## 八、防遗漏机制

### 第一层：文档骨架先行
每份文档第一次创建时，先建完整章节骨架，`[待 Round X 补充]` 标记空洞。

### 第二层：渐进产出地图（PPM）
每轮结束时更新的全局进度追踪表。下一轮开始时先过一眼，红色和黄色就是遗漏信号。

### 第三层：完整性扫描
Round 5 冻结审计轮硬门槛。扫描内容：
- 7 层文档中标记为"本轮需要"的目录，不能有空文件
- 6 核心文档缺一阻塞（v3.0 从 5 扩展为 6，加 AI Agent Prompt 规范）
- 每条链路接口设计含路径/方法/入参/出参/错误码
- 每条链路 todo 每器官含测试点标注
- PPM 中所有进行中的项必须全部转为完成
- 任何文档出现待补充标记且涉及本轮必产内容，阻塞

### 第四层：制衡触发权外移
制衡条款（分叉暂停、时间盒超时、必须确认项）的触发权交给非执行方：
- 分叉暂停：由独立验收角色或 MCP `pause_for_user` 触发
- [x] 证据强制：由 `mark_checked` tool 校验
- 必须确认项（§4.4）：由调度方在启动前与用户确认

在 MCP 落地前靠角色分离 + 人工验收；MCP 落地后靠 tool。

## 九、核心规则

- 声称通过必须附带 pytest 输出
- 关键路径（Dify / 跨链路 HTTP / LangChain / SSE）禁用 mock
- 版本锁定，禁止 "latest"
- 文档 7 层目录结构 + `_chains/` 链路隔离区
- 后端按业务对象分文件，禁止按链路号分
- 5 维度审计中维度 2（动态测试）必跑，不允许跳过
- 代码覆盖率不低于 70%，不允许放宽阈值
