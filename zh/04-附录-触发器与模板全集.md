## E. 反向流程总表

| 场景 | 发生轮次 | 处理 |
|------|---------|------|
| 目录结构不合理 | Round 0 | 回 Round 0 重做 |
| 需求矛盾 | Round 1a | 找业务方确认，修改用户故事 |
| 链路重叠 | Round 1b | 回 Round 1a 重新分类 |
| MVP 模糊 | Round 1b | canyu 拍板 |
| 选型后跑不起来 | Round 2a | 排查版本，不轻易换技术栈 |
| 目录结构不合理 | Round 2b | 重新设计，不要在错的结构上写代码 |
| 契约变更 | Round 2b+ | 回 Round 2b 重新冻结契约 |
| 规范执行不下去 | Round 3 | 不重新设计，用例外记录留 Round 8 复盘 |
| 表结构跟规范不一致 | Round 4a | 重新对齐 |
| 记忆架构无隔离 | Round 4a | 重新设计 user_id 隔离方案 |
| 骨架跑不起来 | Round 4b | 排查 config / 依赖 / 版本，不重写架构 |
| Agent 框架缺失 | Round 4b | 补构造 Agent 框架，不跳过 |
| 硬冻结决策有误 | Round 5 | 回对应轮次重讨论，受影响产出物降级 |
| 单链路失败 | Round 6 | 隔离该链路调试，不破坏骨架 |
| 整合冲突 | Round 7 | 隔离区（t2）解决，不动主分支 |
| 跨链路 HTTP 失败 | Round 7 | 拉 1 个组员在隔离区复现 |
| 严格类未 100% | Round 7/8 | 修代码，不允许发版 |
| 发版失败 | Round 8 | 回退 + 重新整合，不直接再发 |
| 后期发现某轮硬冻结决策有误 | 任意轮 | 标红，回该轮重讨论，后续产出物不自动作废，协调方判断影响范围 |
