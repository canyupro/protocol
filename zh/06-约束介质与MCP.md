# 06 — 约束介质与 MCP

> 规程的约束按强约束度分三层。本文定义三层、迁移判据、MCP tool 规格、声明卡与验收锁。

---

## 一、约束介质分层

### 1.1 为什么分层

文本规则对弱模型软约束失效——模型可不读、读了可不遵循、遵循了可软违反。按强约束度分层，把"违反后果严重 + 文本层反复失效"的规则迁到工具层，实现与模型强度解耦的硬约束。

### 1.2 三层定义

| 层 | 介质 | 适用规则 | 强约束度 |
|---|---|---|---|
| L0 文本 | 规则文档 / 触发器 / 模板 | 流程性、项目特定、低风险 | 低（软约束） |
| L1 协议 | 模式声明卡 / 产出契约 / 验收 checklist | 跨轮次行为约束 | 中 |
| L2 工具 | MCP tool（不调用不能产出） | 制衡类、证据类、暂停类 | 极高（硬约束） |

### 1.3 迁移判据

- 一条规则若**违反后果严重 + 文本层反复失效** → 迁往 L2
- 若**违反后果中等 + 需跨轮次约束** → 迁往 L1
- 其余留 L0

## 二、L0 文本层

### 2.1 适用范围

- 流程性规则：Round 0-8 各轮产出物、每轮输入输出
- 项目特定规则：必须确认项（§4.4）、业务对象命名
- 低风险规则：命名约定、文档结构、格式要求

### 2.2 L0 规则清单

| 规则 | 内容 |
|---|---|
| 52+ 条触发器 | Round 1b-8 全量（见 04-附录 A） |
| 5 维度审计 | 维度 2（动态测试）必跑，不允许跳过 |
| 启动门 8 项 | 硬门槛（见 02 号文件） |
| 三档冻结 | 硬冻结 / 软冻结 / 草稿 + 判定规则（见 00-总纲 §七） |
| 防遗漏三层 | 文档骨架先行 / PPM / 完整性扫描（见 00-总纲 §八） |
| 核心规则 | pytest 输出 / 关键路径禁 mock / 版本锁定 / 7 层文档 / 业务对象命名 / 覆盖率 70% |
| 21 条条款 | §4.1-§4.21（见 00-总纲 §十索引） |

### 2.3 L0 的局限

弱模型可不读、读了可不遵循、遵循了可软违反。对弱模型无硬约束力。需 L1/L2 补充。

## 三、L1 协议层

### 3.1 模式声明卡（§4.4）

**规则**：每轮 AI 首句输出：

```
[模式] caveman={ON|OFF} | §8={ON|OFF} | timebox={N}min | 风险={LOW|MEDIUM|HIGH}
```

**所有平台保留。** 声明卡与继承介质（07 号文件）是正交互补关系：

| 机制 | 作用域 | 方向 | 解决什么 |
|---|---|---|---|
| 继承介质 | 跨会话 | 单向状态读取 | "我是谁、做到哪了"——接手时初始定位 |
| 声明卡 | 会话内每轮 | 双向（模型声明→用户可拒收→模型重答） | "这一轮我打算这么干，对不对"——每轮持续校准 |

**为什么不可省略**：
- 分叉是单向的，主会话无法中途双向校准
- 子智能体独立上下文，不继承主会话历史，每轮仍需校准
- 声明卡是唯一提供"每轮可交互确认"的机制

**用户判断标准**：
- 看到声明卡 → 模型已上线
- 未声明 → 用户可立即指出"模式未声明"，AI 须重答
- 声明与实际不符 → 用户可拒收输出

### 3.2 产出契约

不声明自己开了什么模式，而是产出本身必须满足可校验结构（首句格式、字数上限、必含字段），用产出反推合规。

适用场景：声明卡的补充——声明卡是模型自我声明（可软违反），产出契约是产出本身的结构约束（可机器校验）。

### 3.3 交付一致性验收锁（§4.2）

任何里程碑交付报告必须过 V1-V10 验收 checklist：

| 编号 | 检查项 | 通过标准 |
|---|---|---|
| V1 | 首句模式声明卡 | 存在且四个字段齐全 |
| V2 | commit hash 真实 | `git log --oneline` 能查到 |
| V3 | commit 顺序合理 | 代码 commit 早于报告 commit |
| V4 | commit message 规范 | `feat/fix/docs/chore: <范围> <主题>` |
| V5 | 修改文件在禁区外 | 对照入场卡的禁区清单 |
| V6 | 关键代码位置能 grep 到 | 抽查 3-5 个核心函数名 |
| V7 | pytest 输出真实存在 | 报告里的行数 = 实际测试文件里的 test 函数数 |
| V8 | ruff All checks passed | 未通过必须在遗留问题里显式说明 |
| V9 | AC 逐条有证据 | 不接受"应该通过""看起来 OK" |
| V10 | 遗留问题段存在 | "无"也算存在 |

**核心原则**：交付报告是被验收的对象，不是单方声明。调度方只信 `git log` + 命令实际输出。

**常见口径漂移反例**：

| 反例 | 修正 |
|---|---|
| commit hash 不更新 | 写报告最后一步必须重新 `git rev-parse HEAD` |
| pytest 未跑即报完成 | 打 ✅ 的测试类任务必须有可粘贴的命令输出 |
| 混淆 implementation done 与 user-verified | AC 只能在测试实际跑绿或 review 通过后才打 ✅ |
| 截图未纳入版本 | 所有证据必须在仓库内 |
| SSE 事件字段"应该没变" | 必须新增断言测试 |
| 向后兼容口头声明 | 必须靠前序里程碑测试仍全绿证明 |

### 3.4 L1 的局限

L1 仍是"模型可软违反"的约束——声明卡可以写了不执行，产出契约可以绕过。L1 比 L0 强（可机器校验），但不如 L2（L2 不调用不能产出）。

## 四、L2 工具层（MCP）

### 4.1 设计原则

1. **最小化**：只承接文本层反复失效的规则，不把所有规则都 tool 化
2. **可校验**：每个 tool 的参数必须是机器可校验的（URL 可达、字段非空、枚举合法）
3. **不判断**：tool 不做业务判断，只做机械校验。判断仍由人
4. **与模型无关**：tool 是 MCP 协议，任何支持 tool calling 的模型都能用

### 4.2 L2 规则清单

| 规则 | 对应 tool | 强制点 | 状态 |
|---|---|---|---|
| [x] 必带证据 | `mark_checked` | 无证据 reject | ✅ 已实现 |
| 分叉暂停 | `pause_for_user` + `resume_from_pause` | 默认决策被禁 | ✅ 已实现 |
| 报告模板 | `report_step` | 不调用不能 commit | ✅ 已实现 |

### 4.3 tool 规格

#### mark_checked

```python
@mcp.tool()
def mark_checked(
    item_id: str,           # AC 编号 / 任务编号
    evidence_url: str,      # 证据路径
    evidence_type: str,     # pytest | ruff | git_log | screenshot | review
) -> dict:
    """标记 AC / 任务为完成。必须附带证据。无证据或证据不可达 → reject。"""
```

校验逻辑：
- `pytest`：校验测试函数在文件中存在
- `git_log`：校验 commit hash 真实（`git rev-parse --verify`）
- `ruff`：校验文件存在
- `screenshot`：校验文件在 docs/ 下
- `review`：不做机器校验（人工 review 无法机器验证）

对应规程：验收锁 V7/V9 自动化。

#### pause_for_user + resume_from_pause

```python
@mcp.tool()
def pause_for_user(
    forks: list[dict],      # [{"option": "继续修", "risk": "MEDIUM"}, ...] 至少 2 项
    context: str,           # 为什么需要暂停
) -> dict:
    """策略分叉点强制暂停，把决策权交还用户。"""

@mcp.tool()
def resume_from_pause(
    pause_id: int,          # pause_for_user 返回的 pause_id
    choice: str,            # 用户选择的 option 文本
) -> dict:
    """用户做出分叉选择后恢复执行。"""
```

两段式语义（不阻塞线程）：
- pause：校验 forks ≥ 2 + risk 合法 → 记录 SQLite(PAUSED) → 返回分叉选项
- resume：校验 pause_id 存在且 PAUSED + choice 在 forks 内 → 更新 SQLite(RESUMED)

为什么不清阻塞线程：MCP stdio server 是单线程的，阻塞会卡死。tool 做校验和记录，不做强制拦截——"AI 不继续产出"靠客户端/规程层保证。

对应规程：§4.6 时间盒分叉暂停 tool 化。

#### report_step

```python
@mcp.tool()
def report_step(
    phase: str,             # read | plan | implement | test | wrap_up
    content: str,           # 本步骤实际产出摘要，≤500 字
    artifacts: list[str],   # 产出文件路径列表
) -> dict:
    """报告规程步骤完成。不调用本 tool 不允许进入下一步 / 不允许 commit。"""
```

校验逻辑：
- phase 在枚举内
- content 非空 + ≤ 500 字（防止决策成本转嫁）
- artifacts 非空 + 路径存在

对应规程：报告模板压缩，削减噪声。

### 4.4 接入方式

MCP server 配置（以 ZCode 为例）：

```json
{
  "protocol-mcp": {
    "command": "<python_path>",
    "args": ["-B", "-m", "mcp_server.server"],
    "cwd": "<project_root>"
  }
}
```

重启 AI 客户端后生效。tool 调用历史记录在 SQLite（`mcp_server/state.db`）。

### 4.5 已知限制

| 限制 | 说明 |
|---|---|
| tool 不强制阻塞 | pause_for_user 返回 paused=True 后，AI 是否真停止靠客户端/规程层 |
| 无强制调用机制 | AI 可以不调用 tool 直接标完成；需客户端层强制 |
| stdio 单进程 | 多对话共享同一 server，SQLite 并发写入可能冲突（低风险） |
| cwd 硬编码 | 配置里 cwd 写死项目路径，换项目需改配置 |

## 五、声明卡与继承介质的关系

### 5.1 正交互补

声明卡（L1）和继承介质（07 号文件）解决不同问题，不可互相替代：

| 维度 | 声明卡 | 继承介质 |
|---|---|---|
| 作用域 | 会话内每轮 | 跨会话/跨智能体 |
| 方向 | 双向（模型声明→用户可拒收→重答） | 单向状态读取 |
| 解决什么 | "这一轮我打算这么干，对不对" | "我是谁、做到哪了" |
| 可交互 | 是 | 否（分叉单向/子智能体独立上下文） |

### 5.2 为什么不把声明卡迁到 L2

声明卡的本质是"模型自我声明 + 用户双向确认"——需要一个人类在回路里拒收/重答。L2 tool 做的是机械校验（证据是否齐全/是否在分叉点），不做"这一轮方向对不对"的判断。判断仍由人，所以声明卡留在 L1。
