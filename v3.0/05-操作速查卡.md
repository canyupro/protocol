## G. Runbook 速查（§4.1）

### 每次会话开始前 3 秒自检

- [ ] Docker 已启动？
- [ ] 上次改了 .py 吗？改了就先清 `__pycache__`
- [ ] 后端用 `python -B scripts\run_backend.py`，不用 `uvicorn`

### 常见故障排查表

| 症状 | 首查项 | 二查项 | 兜底 |
|---|---|---|---|
| 登录"网络错误" | Docker 是否启动 | `docker compose ps` 是否 healthy | 看后端日志 |
| Psycopg ProactorEventLoop | 是否用 uvicorn 直接起 | 改用 run_backend.py | 检查 SelectorEventLoopPolicy |
| 改了 .py 表现像没生效 | __pycache__ 是否清 | 是否加 -B | 重启后端 |
| 中文乱码 ??? | .pyc 缓存 | LLM provider 是否切对 | .env LLM_PROVIDER |
| LangChain 返回空 content | V4 thinking 是否禁用 | model_name 是否正确 | 见技术文档 |
| RAG 检索超时 | Milvus 是否 healthy | embedding 是否可达 | timeout 3s 内空返回属正常降级 |
| pytest event loop 报错 | async fixture 冲突 | 重复 import | 见测试规范 |

## H. 验收锁速查（§4.2）

调度方收到交付报告后，按此表逐条核对。任一不通过 → 退回。

| 编号 | 检查项 | 通过标准 |
|---|---|---|
| V1 | 首句模式声明卡 | 存在且四个字段齐全 |
| V2 | commit hash 真实 | `git log --oneline` 能查到 |
| V3 | commit 顺序合理 | 代码 commit 早于报告 commit |
| V4 | commit message 规范 | `feat/fix/docs/chore: <范围> <主题>` |
| V5 | 修改文件在禁区外 | 对照白名单 |
| V6 | 关键代码 grep 到 | 抽查 3-5 个核心函数名 |
| V7 | pytest 输出真实 | 行数 = 实际 test 函数数 |
| V8 | ruff All checks passed | 未通过必须在遗留问题说明 |
| V9 | AC 逐条有证据 | 不接受"应该通过" |
| V10 | 遗留问题段存在 | "无"也算存在 |

## I. MCP 速查

| 场景 | 调用 | 说明 |
|---|---|---|
| 标记 AC 完成 | `mark_checked("AC-F20-1", "tests/test_k12_facts.py::TestHistoryRecallIntent", "pytest")` | 无证据 reject |
| 策略分叉暂停 | `pause_for_user([{"option":"继续修","risk":"MEDIUM"},{"option":"降级","risk":"LOW"}], "测试失败原因不明")` | forks ≥ 2 |
| 用户选择后恢复 | `resume_from_pause(pause_id, "继续修")` | choice 必须在 forks 内 |
| 报告步骤完成 | `report_step("implement", "实现 mark_checked", ["mcp_server/server.py"])` | content ≤ 500 字 |

evidence_type: `pytest` / `ruff` / `git_log` / `screenshot` / `review`

phase: `read` / `plan` / `implement` / `test` / `wrap_up`
