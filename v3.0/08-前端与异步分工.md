# 08 — 前端与异步分工

> 前端完成定义、视角收束、单源 View 模板、异步分工协议、执行方先行模式。

---

## 一、前端完成定义（§4.5）

### 1.1 核心原则

**前端 = 交互层，不是展示层。** 「可演示 ≠ 可交互」。任何不能被点击 / 输入 / 触发请求 / 看到结果反馈的页面，不视为合格前端。

### 1.2 完成定义

以项目需求为中心、角色视角为入口、覆盖 80-90% 后端能力、低设计复杂度。

### 1.3 每个UI元素必答三问

- 这是谁会用？（角色 / 视角）
- 用它能干什么？（动作 / 业务结果）
- 干完之后他能看到什么？（反馈 / 状态 / 后续路径）

任一问题无法回答 → 删除、合并或下沉为后台脚本。

### 1.4 反例

- 在 `/admin` 直接放"学生生成规划"按钮，而 `/student` 没有该入口
- 用一个 `FeatureGrid` 同时给所有角色看，文案靠 if 判断
- 演示导演台只是文字时间线，不指向任何真实接口或真实交互
- 把"可点击但点了什么都没发生"的按钮当作前端能力
- 收束阶段仍把测试用的"调试入口"留在生产路径上

## 二、视角收束

### 2.1 测试阶段：管理界面集中堆放

功能未完整、视角拆分未冻结前，允许使用 `/dev`（或专门的内部 dev 页）作为"总入口"，把各种新功能先一股脑挂上去。

允许：
- 看到任意角色的功能入口
- 调用任意角色的接口（仍需后端权限校验）
- 展示原始数据 / 调试信息

但必须显式标注 `测试入口 / dev only / 收束后下线`。

### 2.2 收束阶段：按视角拆分界面

当任一条件触发，进入 UI 收束阶段：
- 用户明确说"准备演示 / 准备上线 / 准备面试"
- 进入 Round 7 / Round 8 或对应的 UI 收束轮次
- 进入浏览器级演示验收之后

收束阶段必须执行：

1. 对照角色矩阵，列出当前在管理界面里的每一个功能，标注真实归属角色
2. 把每个功能迁移或复刻到对应角色页：
   - 学生功能 → `/student/*`
   - 家长功能 → `/parent/*`
   - 老师功能 → `/teacher/*`
   - 管理员功能 → `/admin/*` 只保留运营 / 权限 / 政策维护类
   - 访客功能 → 公开页 `/`、`/policies`、`/policies/[id]`
3. 管理界面只保留管理员视角真实需要的能力
4. 任一功能在多个角色都需要时，必须按视角做不同的呈现
5. 拆分完成后，必须做一次真实浏览器走通，每个角色至少完成 1 个真实交互动作（点击 → 接口 → 反馈）

## 三、单源 View 模板

### 3.1 三条铁律

1. **真源单写**：业务组件只存在 `app/_shared/views/*` 一份。角色路由 `page.tsx` 是薄壳，只 `return <XxxView />`。任何对业务的改动都改 `_shared/views/*`，不允许在路由 `page.tsx` 写业务
2. **角色态共享**：使用一个轻量 `RoleContext` 持有当前角色与共享 token，不写 `localStorage`；测试期由 `/dev` 的 `RoleSwitcher` 覆盖 Context，生产期由真实登录态接管
3. **生产期下线**：以 env 开关 + middleware + 主导航条件渲染三层下线测试入口；生产构建未开启开关时，`/dev/**` 直接 404

三条铁律必须同时满足，缺一条就退化为常见反模式。

### 3.2 目录结构模板

```
frontend/
  app/
    _shared/
      role-context.tsx           # RoleProvider / useRole / isDevConsoleEnabled / APP_ROLES
      DevConsoleNavLink.tsx      # 主导航条件 dev 入口；env=0 时返回 null
      views/
        StudentView.tsx          # 学生真源 View
        ParentView.tsx           # 家长真源 View
        TeacherView.tsx          # 老师真源 View
        AdminView.tsx            # 管理员真源 View
    student/page.tsx             # 薄壳：return <StudentView />
    parent/page.tsx              # 薄壳
    teacher/page.tsx             # 薄壳
    admin/page.tsx               # 薄壳
    dev/page.tsx                 # /dev 聚合入口：仅 import _shared/views/*，不写第二份业务逻辑
    layout.tsx                   # 包裹 RoleProvider + DevConsoleNavLink
  middleware.ts                  # matcher: /dev, /dev/:path*；env=0 时 rewrite -> /not-found
.env.example                     # 显式声明 NEXT_PUBLIC_ENABLE_DEV_CONSOLE
```

关键点：
- `app/_shared/` 以下划线开头，不会被 Next App Router 暴露为路由
- 路由薄壳 `page.tsx` 不允许任何业务逻辑、`useState`、`useEffect`、接口调用
- `dev/page.tsx` 只允许做三件事：渲染顶部 dev banner、维护 `RoleSwitcher`、按当前 role 渲染对应 View

### 3.3 采用条件

满足以下任一条件即建议落地：
- 项目存在 ≥3 个固定角色，且各角色都有独立交互页面
- 演示阶段需要在一个入口快速切换不同角色
- 上线前必须强制下线测试入口，避免污染生产路径

如果只有一个角色或仅做营销页，模板会带来过度抽象，可不采用。

### 3.4 验收清单

落地后必须在浏览器中人工通过以下 6 步：

1. 新建 `frontend/.env.local`，写入 `NEXT_PUBLIC_ENABLE_DEV_CONSOLE=1`
2. 启动前端 → 顶部导航出现醒目 `DEV` 按钮
3. 访问 `/dev` → 顶部红色 banner、`RoleSwitcher`、token 输入框全部可见
4. 切到学生 → 粘贴 demo token → 加载学生数据 → 生成规划，确认为真实接口闭环
5. 切家长 / 老师 / 管理员，确认渲染对应真源 View，无第二份业务实现
6. 关闭 env 后重启前端 → `/dev` 返回 404、主导航 `DEV` 按钮不渲染

任何一项未通过，不视为完成本模板落地。

## 四、异步分工协议（§4.3）

### 4.1 分工边界

当主调度方与执行方（弱模型）分离时：

- **调度方**：只做首验 / 终验 / bug 定性
- **执行方**：施工 + 中间验证
- 中间验证下沉执行方，调度方不重跑完整验收，只做抽检

### 4.2 成本纪律

低成本模型完成后，高成本调度方不重跑完整验收，只做抽检。否则分工设计失效。

### 4.3 里程碑粒度

- 单个 M/F 工时建议：2-4h 执行方施工
- 单个 M/F 必须可独立验证（AC ≥ 3 条）
- 跨链路改动多于 2 个模块 → 拆分为多个里程碑
- 涉及必须确认项 → 调度方必须在启动前和用户确认，执行方不主动触碰

### 4.4 必须确认项（§4.4）

以下变更必须调度方在启动前与用户确认，执行方不主动触碰：

- 数据库结构变更
- RBAC 权限矩阵种子数据修改
- 心理健康记录访问权限变更（涉未成年人保护）
- 升学政策数据来源变更
- 白名单 / 红名单条目人工增删

## 五、执行方先行

### 5.1 模式

执行方可在调度方介入前完成施工，调度方只做终验。省掉入场卡编写 + 上下文传递包 + 声明卡。

```
1. 任意模型接手，读继承快照
2. 判定自己是执行方（施工方已完成=否）
3. 直接施工（不需要等入场卡）
4. 完工后更新快照：施工方已完成=是 + commit hash + 验收证据
5. 调度方读快照
6. 判定自己是调度方（施工方已完成=是, 调度方已介入=否）
7. 做终验（只做验收，不重跑施工）
8. 验收通过 → 更新快照 → 下一任务
```

### 5.2 前提

继承快照必须包含足够信息让执行方知道"做什么、怎么做、边界在哪"。这要求快照比传统入场卡更结构化。

### 5.3 平台限制

- **ZCode**：分叉保留上下文 + 子智能体独立模型 → 执行方先行 + 子智能体实时介入（最优）
- **其他平台**：无分叉保留上下文 → 执行方靠继承快照恢复上下文，介入靠快照邮箱人肉桥接

详见 07 号文件。
