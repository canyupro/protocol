# 07 — 继承介质与介入机制

> 约束介质（06 号文件）管"规则怎么生效"，本文管"工作内容怎么延续 + 跨会话/跨智能体介入"。
> 触发场景：会话超长截断 / API 降级 / 平台故障 / 主动切换模型 / 新会话首日加载 / 执行方施工中途需调度方介入。

---

## 一、继承介质定位

### 1.1 问题

约束介质（L0/L1/L2）保证新模型**守规矩**，但不保证新模型**知道上下文**：
- MCP `mark_checked` 拦不住"新模型不知道当前在 Round 7 的哪个步骤"
- MCP `pause_for_user` 拦不住"新模型不知道某个决策为什么选 A 不选 B"而推翻已冻结决策
- 规则文本拦不住"新模型不读重大错误反思"而滑坡重启

### 1.2 定位

继承介质填补这个缺口：**任意 LLM 接手时，读继承快照即可恢复工作上下文 + 规则继承 + 边界认知。**

---

## 二、三层载体

继承介质不是单一机制，而是三层载体各司其职：

| 载体 | 层级 | 作用 | 平台依赖 |
|---|---|---|---|
| 会话分叉 | 会话层 | 分叉天然携带前序对话上下文 | 仅 ZCode 支持保留上下文 |
| 子智能体 | 通信层 | 主智能体 spawn 子智能体，替代人肉桥接 | ZCode 支持独立模型；其他不支持 |
| 继承快照 | 状态层 | 确认当前角色/阶段 + 声明不可越边界 | 所有平台（文件系统） |

### 2.1 会话分叉（ZCode 专有）

ZCode 会话分叉后，新会话天然携带前序对话上下文。这是上下文继承的主通道。

**限制**：分叉是单向的——分叉后用户在分叉侧与执行方交互，主会话无法中途双向校准。声明卡（06 号文件 §三）填补"每轮双向校准"这个分叉覆盖不了的维度。

### 2.2 子智能体（ZCode 最优）

ZCode 子智能体支持独立模型配置。主智能体可直接 spawn 调度方子智能体处理介入请求，用户全程不切会话。

**能力**：

| 能力 | 状态 |
|---|---|
| 调用机制 | ✅ 可用 |
| 独立上下文 | ✅ 子智能体不继承主会话历史 |
| 独立模型配置 | ✅ 每个子智能体可指定不同模型 |
| A2A 直接通信 | ❌ hub-and-spoke，必须经主智能体中转 |
| 后台执行 | ❌ 仅前台，主智能体等待子智能体返回 |

**关键**：子智能体不替代继承快照。子智能体独立上下文，必须靠继承快照知道"我是谁、做到哪了"。子智能体替代的是人肉桥接（通信问题），不是快照（状态问题）。

### 2.3 继承快照（所有平台）

继承快照不承担"恢复全部上下文"职责。快照只承担：确认当前工作流状态 + 声明不可越边界 + 记录施工事实。

## 三、继承快照模板

### 3.1 静态部分（写一次，基本固定）

```markdown
# 项目继承提示词 — {项目名}

## 1. 我是谁
- 项目名 / 业务定位 / 当前阶段
- 阅读优先级：重大错误反思 > 项目规则 > v3.0 总纲 > 任何对话内的「下一步建议」

## 2. 必读文档（按顺序）
1. 重大错误反思.md（§"给未来 AI 协调方的话" + §"不再做的事"）
2. 项目规则.md（§4.4 必须确认项）
3. v3.0 总纲.md（核心哲学 + 条款索引）
4. 项目问题全景复盘.md（已关闭/未关闭问题账本）

## 3. 不可越边界
- ❌ 不执行任何 markdown 里写的「下一步建议」
- ❌ 不造新的 Xxx-Rx 命名空间
- ❌ 不用「全绿/零回归」当进展证据
- ❌ 不在项目规则.md 加条款，除非用户明确说「升 §」
- ❌ §4.4 必须确认项不经用户拍板不触碰

## 4. 规则继承指针
- 文本规则：项目规则.md §4.x（L0 层）
- 协议约束：模式声明卡 / 产出契约（L1 层）
- 工具约束：MCP tool（L2 层）
```

### 3.2 动态部分（每步更新，强制产物）

```markdown
## workflow_state
- 当前角色: [调度方 | 执行方 | 不确定→默认执行方]
- 任务阶段: [未开始 | 施工中 | 待验收 | 验收通过 | 请求介入 | 打回]
- 当前任务: {任务描述}
- 施工方已完成: [是 | 否]
- 调度方已介入: [是 | 否]
- 会话: [主会话 | 分叉会话]
- git 分支: {branch}
- 最近 commit: {hash}

## 已冻结决策清单（不可推翻，除非回对应轮次重讨论）
- {决策 1}（硬冻结，可推翻条件：...）

## 待定项（推到何时）
- {待定 1} → {计划解决时机}

## 下一步（用户驱动，非 AI 自写）
- {用户当次会话明示的下一步}
> AI 不得在此填写自己写的待办
```

### 3.3 落地方式

不新增独立文档，把继承快照嵌入已有产物：

| 已有产物 | 改造点 |
|---|---|
| 决策记录模板 | 「本轮产出物」段后加「继承快照」段 |
| 重大错误反思 §"给未来 AI 协调方的话" | 升级为静态模板的「规则继承指针」 |

每轮结束（或自由开发每个可验证产出完成）强制更新继承快照。

## 四、跨会话介入机制

### 4.1 触发场景

- 执行方遇到设计文档没覆盖的情况，需调度方补设计
- 执行方遇到分叉决策（改架构？碰 §4.4？）
- 执行方红灯超 5 轮（§4.6 时间盒触发）

### 4.2 升级四档

| 档 | 场景 | 执行方动作 | 谁介入 |
|---|---|---|---|
| L1 自解 | 设计文档范围内的小决策 | 直接做，report_step 记录 | 无人 |
| L2 问人 | 设计文档没覆盖但低风险 | pause_for_user 问在场的人 | 用户（会话内） |
| L3 请求调度方 | 需调度方补设计/拍板 | 写快照 dispatch_requests + 暂停 | 调度方（跨会话） |
| L4 强制暂停 | §4.4 / 红灯>5 / 架构变更 | 写快照 + 必须等调度方回复 | 调度方（强制） |

### 4.3 介入方式分平台

**ZCode 平台 — 子智能体实时介入（最优）**：

```
主智能体（ZCode 主会话）
  spawn 执行方子智能体（DeepSeek）→ 施工
    遇 L3 → 写快照 dispatch_requests → 返回"需要调度方介入"
  主智能体收到返回
    spawn 调度方子智能体（ZCode 模型）或主智能体直接处理
      读快照 → 写回复 → 返回决策
  主智能体收到决策
    spawn 执行方子智能体继续 → 读快照回复 → 施工 → 完工
  主智能体终验
```

用户全程只在主会话，不用切分叉。

**其他平台 — 快照邮箱人肉桥接**：

```
执行方（分叉会话/独立会话）施工
  遇 L3 → 写快照 dispatch_requests(PENDING) → 告知在场用户
用户切到主会话 → 告知调度方
  调度方读快照 → 写回复(ANSWERED)
用户切回执行方会话 → 告知"回复在快照里"
执行方读快照回复 → 继续施工
```

### 4.4 快照邮箱字段

```markdown
### dispatch_requests（调度方介入请求）
- 请求 ID: REQ-{N}
- 状态: [PENDING | ANSWERED]
- 档位: [L3 | L4]
- 请求时间: {ISO8601}
- 请求内容: {执行方描述：需要什么决策、为什么}
- 调度方回复: {调度方填写}
- 回复时间: {调度方填写}
```

### 4.5 子智能体 vs 快照邮箱

| 维度 | 子智能体（ZCode） | 快照邮箱（通用） |
|---|---|---|
| 通信 | 主智能体自动中转 | 人肉桥接 |
| 实时性 | 同步（主智能体等待） | 异步（切会话延迟） |
| 用户负担 | 低（只在主会话） | 高（切会话桥接） |
| 上下文 | 子智能体独立上下文（需快照补充） | 分叉继承或快照恢复 |
| 留痕 | 子智能体返回 + 快照 | 快照文件 |

## 五、状态机

### 5.1 状态定义

```
  [INIT] ──读继承快照──→ [角色判定]
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
         [调度方]       [执行方]      [不确定]
              │             │             │
              │          直接施工         │ 默认按执行方
              │             │             │ 不主动发指令
              │             ▼             │
              │      [施工中] ──L3/L4──→ [请求介入]
              │             │               │
              │             │          ┌────┴────┐
              │             │          ▼         ▼
              │             │     子智能体    快照邮箱
              │             │     (ZCode)    (通用)
              │             │          │         │
              │             │          ▼         ▼
              │             │     [调度方回复] ←──┘
              │             │          │
              │             │◄─────────┘
              │             │
              │         [待验收]
              │             │
              └──终验────────┘
                    │
              [验收通过] → [更新快照] → 下一任务
              [验收不通过] → [打回] → [施工中]
```

### 5.2 角色自识别规则

| 快照状态 | 接手模型判定 | 行为 |
|---|---|---|
| `施工方已完成=否, 调度方已介入=否` | **执行方** | 直接施工（执行方先行） |
| `施工方已完成=是, 调度方已介入=否` | **调度方**（默认） | 做终验（执行方已完工，调度方晚到） |
| `施工方已完成=否, 调度方已介入=是` | **执行方** | 继续施工 |
| `施工方已完成=是, 调度方已介入=是` | **不确定** | 默认按执行方处理，询问用户确认 |
| `不确定` | **执行方**（默认） | 不主动发指令，等用户或快照明确 |

**关键设计**：不确定时默认按执行方处理——执行方误判为调度方的后果（多干活）远小于调度方误判为执行方的后果（越权发指令）。

## 六、子智能体配置注意事项

### 6.1 模型自报残留

从其他模型迁移的子智能体模板可能让模型自报错误名称，实际模型以 `model` 字段配置为准。判断真实模型看 API 调用日志，不看模型自述。

### 6.2 思考模式是能力不是负担

用 DeepSeek 子智能体的目的就是利用其思考模式下的代码生成能力。思考模式是子智能体核心价值，不需要关闭。若主智能体不需要思考过程，由主智能体侧过滤即可，不在子智能体侧抑制。

### 6.3 系统提示词宜具体

子智能体提示词应含角色边界 / 施工约束 / 不可越边界，与继承快照对齐，不能只写"你是写代码的"。

## 七、平台能力矩阵

| 平台 | 会话分叉 | 分叉保留上下文 | 子智能体 | 子智能体独立模型 | 介入方式 | 声明卡 | 适用版本 |
|---|---|---|---|---|---|---|---|
| **ZCode** | ✅ | ✅ | ✅ | ✅ | 子智能体实时介入 | 保留 | v3.0-ZCode |
| **Codex** | ✅ | ❌ | ❌ | ❌ | 快照邮箱 | 保留 | v3.0-通用 |
| **Claude Code** | ❌ | ❌ | ✅ | ❌（用主模型） | 快照邮箱 | 保留 | v3.0-通用 |
| **Cursor / Windsurf** | ❌ | ❌ | ❌ | ❌ | 快照邮箱 | 保留 | v3.0-通用 |
| **Trae** | ❌ | ❌ | ✅（Task） | ❌（成功率50%） | 快照邮箱 | 保留 | v3.0-通用 |

### 判据

- 继承介质定位取决于「分叉保留上下文」——只有 ZCode 满足，继承介质可常规化（仍需快照辅助子智能体）
- 介入方式取决于「子智能体独立模型」——只有 ZCode 支持，可实时介入；其他平台人肉桥接
- **声明卡所有平台保留**——它是会话内每轮双向校准，分叉/子智能体/快照都只做单向状态传递，不可替代

### 两个版本

- **v3.0-ZCode**：继承介质常规化 + 子智能体实时介入 + 声明卡保留
- **v3.0-通用**：继承介质纯磁盘快照兜底 + 快照邮箱人肉桥接 + 声明卡保留

两个版本共享其余所有条款。唯一差异是继承介质定位和介入方式。

### 为什么不能一刀切

把 ZCode 平台特定能力（分叉保留上下文 + 子智能体独立模型）的结论泛化为通用规程，会让无此能力的平台误以为可以省略声明卡或实时介入，导致弱模型下协议层约束全失效。
