## Round 6: 链路并行开发

**输入**：骨架 + 启动门已通过 + 上下文传递包（或继承快照）。

**产出物**：每条链路代码 + 单链路测试 + _chains/ 接口设计草稿。

**§4.15 Agent 降级策略**（含 Agent 链路时）：

每个 Agent 链路必须实现降级路径：

| 降级点 | 规则 |
|---|---|
| LLM 失败 | SSE error 事件 + fallback token（温和提示）+ done 仍发送 |
| 工具失败 | 所有工具兼具 fallback 文案，不抛错 |
| RAG 检索失败 | 返回"未找到相关内容"，不阻塞主流程 |
| 向量库不可用 | 主流程仍通过（降级友好） |

降级测试在 Round 7 五维度审计中检查。

**§4.6 时间盒 + 红灯分级**：

| 规则 | 内容 |
|---|---|
| Todo 粒度 | 对应可验证产出，不对应 shell 命令，推荐 5-10 分钟 |
| 普通红灯 | 5 轮才暂停（小测试/脚本/路径问题不频繁打断用户） |
| 高风险红灯 | 2 轮暂停（架构/权限/心理健康/RBAC/政策来源） |
| 单步超时 | 超 10min 必报；长命令超 3min 开始前预告 |
| 自动推进 | 用户已确认策略内默认自动推进；策略分叉/超时/必须确认项才暂停 |

**§4.21 依赖新增流程**：

1. 链路 Agent 发现需要新包 → 报协调方（issue / 决策记录）

### 动态测试规范

**覆盖率阈值**：不低于 70%，不允许放宽，不允许覆盖率报告缺失。

**mock 禁用原则**（关键路径禁用 mock）：

禁止 mock：
- LangChain LLM 调用
- SSE 流式
- 跨链路 HTTP 调用
- 业务核心逻辑

允许 mock：
- 数据库（in-memory SQLite 或专用 test_db schema）
- 第三方 HTTP（如 OpenAI，仅单测，集成必真跑）
- 时间（模拟特定时间点）
- 随机数

**Agent 链路测试补充**：
- 危机短路测试：关键词命中 → 安全模板返回，不走 LLM
- 降级测试：LLM 失败 → error 事件 + fallback
- 安全边界测试：学生端零暴露 / user_id 隔离 / 评级取严
- 降级测试可用 mock（避免真 LLM 9 分钟），但危机短路和真实 LLM 链路完整性由少量"真跑"保证

**测试分层**：

- 单测：1 个函数，1 个文件，无依赖
- 集成：多个模块，1 个 API 端点
- E2E：完整业务流，跨多个端点

**commit message 必须附带 pytest 输出**（最后 5 行）：

```
[链路 1] 实现 customer 注册 API

- model: Customer / schema: CustomerCreate / service: create_customer
- api: POST /api/v1/customers

## pytest 输出
tests/test_chain1_customer.py::test_create_customer_normal PASSED
tests/test_chain1_customer.py::test_post_customers_201 PASSED
3 passed in 0.42s
```

## Round 7: 多链路整合

**输入**：N 条链路各自完成 + 各自的接口设计。

**产出物**：合并后代码 + 集成预演 + 5 维度审计（每条链路 1 份）+ 前端视角收束验收 + 会议纪要。

**关键步骤**：

1. 冲突预防（公共文件拆分方案已拍、命名规范已拍、跨链路调用规范已拍）
2. 隔离区合并：`git checkout -b t2 origin/main`，逐条链路 merge
3. 集成预演：跨链路 HTTP 真实调用（不 mock），端到端跑 1 个完整业务流
4. 冲突处理：不允许暴力覆盖，不允许留 TODO 后续修
5. **前端视角收束**（含前端时）：按 08 号文件 §二 执行——角色拆分 / `/dev` 下线 / 每角色至少 1 个真实交互闭环
6. **E2E 分层**（§4.7）：fast audit（门禁）+ full-ai（演示彩排），慢链路不混入门禁

**§4.8 场景化回归测试集**：

发版前必产场景化回归测试集：
- 假种子用户（f21_ 前缀，结束后清理）
- Fake LLM / RAG monkeypatch（避免真 LLM 慢测）
- 场景编号（S01-SN，覆盖各里程碑能力）
- 跨学生隔离安全测（user_id 强隔离验证）
- TXT / JSON 双输出
- try/finally 自动清理

特别适合 AI Agent 类项目。

**§4.2 交付一致性验收锁**：

发版前交付报告必须过 V1-V10 验收 checklist（详见 06 号文件 §三）。核心原则：交付报告是被验收的对象，不是单方声明。

**关键步骤**：

1. E2E 测试：真实跑（不 mock），覆盖完整业务流
2. **场景化回归测试集**（§4.8）：假种子 + mock + 场景编号 + 双输出 + 清理
3. 覆盖率最终检查：`pytest --cov=app --cov-fail-under=70`，不允许放宽
4. **评测分级确认**（§4.19）：严格类 100% + 弹性类按阈值
5. 写测试报告（汇总 5 维度审计 + E2E + 场景化回归结果）
6. 写验收标准对照（逐项验收，跟 Round 1b 的验收标准对齐）
7. **验收锁 V1-V10**（§4.2）：交付报告过 10 项检查
8. **继承快照终版**（§4.12）：更新 workflow_state = 验收通过 + 最近 commit + 已冻结决策清单
9. 性能测试（可选）：Locust 跑 P95 低于 500ms
10. 上线 checklist：


## 给协调方的提示词设计守则

### 原则 1：硬约束放在提示词最后

AI 最关注末尾内容，硬约束必须放在显眼的末尾位置。

### 原则 2：执行检查项必须可验证

| 不可验证 | 可验证 |
|---------|--------|
| "要测试" | "必跑 pytest tests/... -v，带输出 commit" |
| "要写接口文档" | "接口设计 含路径/方法/入参/出参/错误码" |
| "遵守命名规范" | "snake_case / 复数表名 / 单数模型名" |
| "覆盖率要高" | "覆盖率不低于 70%，不降级" |

### 原则 3：禁止项必须显式列出

AI 默认不知道"不做什么"。

### 原则 4：输出格式必须结构化

AI 输出才能被检查。

### 约束平衡经验

| 类别 | 数量建议 | 例子 |
|------|---------|------|
| 硬约束（必遵守） | 5-12 条/阶段 | mock 禁用 / 锁版本 / 覆盖率不低于 70% / Agent 降级 |
| 建议（AI 自选） | 3-5 条/阶段 | 索引建议 / 错误处理 |
| 禁止项（必显式） | 3-7 条/阶段 | 禁改 core / 禁 mock 关键 / 禁声称通过 / 禁自行 pip install |

违反硬约束或禁止项：阻塞，标红。违反建议：警告，标黄。
