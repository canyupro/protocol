> 实验时间：2026-07-03
> 实验载体：MCP server 开发（mark_checked / pause_for_user / report_step 三 tool）
> 实验平台：**ZCode**（会话分叉 + 保留上下文）
> 分支：feature/protocol-mcp（已合并回 main）
> 状态：**实验完成，三假设在 ZCode 平台全部成立**

## 一、实验目标

验证三个假设：

| 假设 | 内容 | 当前痛点 |
|---|---|---|
| H1 角色自识别 | 接手模型读继承快照即可判断自己是调度方还是执行方，不需要外部指派 | v2.0 靠调度方显式发入场卡指派角色 |
| H2 声明卡可省略 | 继承快照足够完整时，新模型不需要每轮首句输出模式声明卡 | 声明卡自我声明不可验证 + caveman 不好用（C5 A3） |
| H3 执行方先行 | 执行方可在调度方介入前完成施工，调度方只做终验 | 当前分工要求调度方先发入场卡，执行方才能开工 |

## 二、实验设计

### 三层载体

| 载体 | 层级 | 作用 |
|---|---|---|
| ZCode 会话分叉 | 会话层 | 分叉天然携带前序对话上下文——上下文继承主通道 |
| 继承快照（磁盘文件） | 状态层 | 确认当前角色/阶段 + 声明不可越边界——轻量硬状态 |
| git 版本控制 | 产物层 | 施工代码 commit 留痕 + 回滚能力 |

### 状态机

接手模型读继承快照的 `workflow_state` 段，按 `施工方已完成` × `调度方已介入` 两个布尔值判定角色。不确定时默认按执行方处理（误判后果更小）。

### 三阶段递进

| Phase | 场景 | 复杂度 | 验证点 |
|---|---|---|---|
| 1 | 单 tool 无状态（mark_checked） | 低 | 基础角色识别 + 声明卡省略 |
| 2 | 有状态联动 tool（pause_for_user + resume_from_pause） | 中 | 追加不替换 + 状态流转 + 回归风险 |
| 3 | 既有代码清理 + 报告压缩（report_step + Phase 2 遗留清理） | 高 | 安全清理 + D5 噪声削减 + 三阶段累积无回归 |

## 四、关键发现

### 发现 1：分叉继承替代了声明卡

DeepSeek 在三个 Phase 中从未输出模式声明卡，直接读快照施工。原因：
- 上下文由 ZCode 分叉天然继承（不是从零读文件）
- 状态由快照 workflow_state 确认（磁盘不会软违反）
- 不需要模型嘴里声明"我开了什么模式"

**这直接回应了 C5 A3**：声明卡不管用的根因是"自我声明不可验证"。继承介质把"声明"变成"继承"——状态从分叉+磁盘来，不从模型嘴里来。

### 发现 2：执行方先行省掉三层开销

对比 v2.0 异步分工协议（§4.17），三轮累计省掉：
- 入场卡编写（4 份文档 × 3 阶段 = 12 份）
- 上下文传递包（AGENTS.md / 职责划分 / 开发规则等 × 3 阶段）
- 声明卡（每轮首句 × 全部轮次）

调度方工作量从「写设计 + 写入场卡 + 发指令 + 中间验收 + 终验」降为「写设计 + 终验」。

### 发现 3：追加不替换 + 既有代码清理都能安全执行

Phase 2 改了 models/store/server 三个既有文件但没删任何 mark_checked 代码（14→25 无回归）。Phase 3 不仅追加了 report_step，还安全清理了 Phase 2 遗留（函数内 import 提顶 + 删死代码 init_pause_db），25→36 无回归。

**DeepSeek 正确理解了"追加不替换"和"清理不破坏"两个约束**——这两个约束都只写在设计文档里，靠继承快照 + 分叉上下文传递，没有显式指派。

### 发现 4：角色模糊性是特性不是 bug

执行方继承上下文后会"觉得自己是调度方"——实验证明这不是问题：
- 不确定时默认按执行方处理（§2.3 规则）
- 执行方误判为调度方的后果（多干活）远小于调度方误判为执行方的后果（越权发指令）
- 三阶段角色越权次数 = 0

## 五、实验结论（ZCode 平台）

### 三假设在 ZCode 平台全部成立

| 假设 | 结论 | 证据强度 |
|---|---|---|
| H1 角色自识别 | ✅ 成立 | 3/3 准确，0 越权 |
| H2 声明卡省略 | ✅ 成立 | 3/3 省略，快照完整即满足 |
| H3 执行方先行 | ✅ 成立 | 3/3 执行方独立完工，调度方只终验 |

### 对 v3.0 规程的含义（分平台）

1. **继承介质在 ZCode 平台可常规化**：v3.0 §4.12 / §七 原定位为模型切换时的兜底。ZCode 实验证明它在常规开发中有效，ZCode 平台可升级为主线机制（子智能体独立上下文仍需快照辅助）。

2. **声明卡所有平台保留**：声明卡是会话内每轮双向校准，继承介质是跨会话单向状态传递，正交互补不替代。子智能体独立上下文更需要声明卡校准。~~原"ZCode 可省略"结论已修正。~~

3. **入场卡在 ZCode 平台可大幅简化**：v2.0 §4.17 入场卡（4 份文档）可简化为继承快照的 `workflow_state` + `已冻结决策` + `不可越边界` 三段。

4. **MCP L2 层有效**：三 tool 在实验中作为施工对象验证了可施工性。已接入 ZCode 客户端。

5. **介入机制分平台**：ZCode 子智能体实时介入（最优）/ 其他平台快照邮箱人肉桥接。详见 §九。

### 失败判据复核

实验设计里定义了三个失败判据，均未触发：
- ❌ 角色自识别准确率 < 80% → 实际 100%
- ❌ 越权次数 > 0 且无法靠快照约束 → 实际 0
- ❌ 终验通过率 < 50% → 实际 100%

## 八、平台适用性（⚠️ 不可外推）

### 为什么结论不能泛化

本实验的成功依赖 ZCode 的「会话分叉 + 保留上下文」能力。这是平台特定能力，不是通用能力：
- 上下文继承的主通道是 ZCode 分叉天然携带前序对话，不是靠读磁盘文件恢复。
- 其他平台无此能力时，继承介质退化为纯磁盘快照恢复，无法替代声明卡的"上线确认"功能。
- 把 ZCode 平台特定结论泛化为通用规程，会让无此能力的平台误以为可以省略声明卡，导致弱模型下协议层约束全失效。这违反 D3（规则文件真实性失守）。

### 平台能力矩阵

| 平台 | 会话分叉 | 分叉保留上下文 | 子智能体 | 子智能体独立模型 | 介入方式 | 声明卡 | 适用版本 |
|---|---|---|---|---|---|---|---|
| **ZCode** | ✅ | ✅ | ✅ | ✅（实测确认） | 子智能体实时介入 | 保留 | v3.0-ZCode |
| **Codex** | ✅ | ❌ | ❌ | ❌ | 快照邮箱 | 保留 | v3.0-通用 |
| **Claude Code** | ❌ | ❌ | ✅ | ❌（用主模型） | 快照邮箱 | 保留 | v3.0-通用 |
| **Cursor / Windsurf** | ❌ | ❌ | ❌ | ❌ | 快照邮箱 | 保留 | v3.0-通用 |
| **Trae** | ❌ | ❌ | ✅（Task） | ❌（成功率50%） | 快照邮箱 | 保留 | v3.0-通用 |

### v3.0 两个版本

- **v3.0-ZCode**：继承介质常规化 + 子智能体实时介入 + 声明卡保留。本实验验证。
- **v3.0-通用**：继承介质纯磁盘快照兜底 + 快照邮箱人肉桥接 + 声明卡保留。适用于无分叉保留上下文或无子智能体独立模型的平台。

两个版本共享 v3.0 的其余所有条款（流程骨架/约束介质分层/制衡第四层/MCP L2 等）。**声明卡所有平台保留**——它是会话内每轮双向校准，分叉/子智能体/快照都只做单向状态传递，不可替代。

## 九、子智能体介入机制（2026-07-03 补充）

### 盲区发现

状态机实验三阶段验证了 H1/H2/H3，但发现一个盲区：**分叉后执行方施工中途需调度方介入**时，分叉是单向的，canyu 需人肉桥接。声明卡的可交互性（每轮双向）与分叉的不可交互性（单向状态传递）是正交的，不可互相替代。

### ZCode 子智能体能力（文档+实测）

| 能力 | 状态 | 证据 |
|---|---|---|
| 子智能体调用机制 | ✅ 可用 | codeman 测试：Agent 工具成功启动，读文件+返回结果 |
| 独立上下文 | ✅ 确认 | codeman 不继承主会话历史，自己读文件完成任务 |
| 独立模型配置 | ✅ 已确认生效 | codeman 配置 deepseek-v4-pro，API 侧无 Claude key，实际跑的是 DeepSeek。自报"Claude"是子智能体模板从 Claude 迁移的提示词残留，非模型未生效 |
| 思考模式控制 | ⚠️ 无配置项 | 子智能体无控制 thinking 开关的字段，DeepSeek V4 Pro 可能默认开 thinking 导致响应慢/content 空（同项目 LLM-3 问题）。需在系统提示词内手动约束 |
| A2A 直接通信 | ❌ 不支持 | hub-and-spoke，必须经主智能体中转 |
| 后台执行 | ❌ 仅前台 | 主智能体等待子智能体返回 |

### 子智能体配置注意事项

1. **模型自报残留**：从 Claude 迁移的子智能体模板可能让模型自报"Claude"，实际模型以 `model` 字段配置为准。判断真实模型看 API 调用日志，不看模型自述。
2. **思考模式是能力不是负担**：子智能体用 DeepSeek V4 Pro 的目的就是利用其思考模式下的代码生成能力。思考模式是子智能体的核心价值，不需要关闭。若下游消费者（如主智能体）不需要思考过程，由主智能体侧过滤即可，不在子智能体侧抑制。
3. **系统提示词宜具体**：codeman 当前提示词仅"你是写代码的"过于简短，应补充角色边界/施工约束/不可越边界，与继承快照的 `不可越边界` 段对齐。

### 介入方式

**ZCode（子智能体实时介入）**：主智能体 spawn 执行方子智能体（DeepSeek）施工 → 遇 L3 写快照 → 返回主智能体 → 主智能体 spawn 调度方子智能体处理 → 返回决策 → 主智能体 spawn 执行方继续。canyu 全程不切会话。

**其他平台（快照邮箱人肉桥接）**：执行方写快照 dispatch_requests(PENDING) → canyu 切主会话告知调度方 → 调度方写回复(ANSWERED) → canyu 切回执行方 → 执行方读回复继续。

### 升级四档

| 档 | 场景 | 执行方动作 | 谁介入 |
|---|---|---|---|
| L1 自解 | 设计文档范围内小决策 | 直接做，report_step 记录 | 无人 |
| L2 问人 | 设计文档没覆盖但低风险 | pause_for_user | 用户（会话内） |
| L3 请求调度方 | 需调度方补设计/拍板 | 写快照 dispatch_requests + 暂停 | 调度方（跨会话） |
| L4 强制暂停 | §4.4 / 红灯>5 / 架构变更 | 写快照 + 必须等回复 | 调度方（强制） |

详见 v3.0 总纲 §7.5。
